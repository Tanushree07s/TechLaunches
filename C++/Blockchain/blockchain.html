What Is a Blockchain ?

  • Blockchain is a type of distributed ledger technology (DLT) It records transactions or data across a network of computers (nodes), rather than relying on a central server.
  • All participants hold a copy of the digital ledger.
  • Once added, data is immutable—it cannot be changed or deleted.

Blockchains & Smart Contracts

  • Smart Contracts
    • Smart contracts are self-executing digital contracts stored on a blockchain.
    • A smart contract is coded with predefined "if-then" rules and deployed to the blockchain, becoming public, immutable, and ready to respond.
    • A user initiates a transaction (e.g., sends payment), which the smart contract continuously watches for on the blockchain.
    • If the contract’s conditions are met, it auto-executes the specified action (e.g., sends an asset). If not, it remains inactive.
    • All interactions and outcomes are permanently recorded on the blockchain, allowing anyone to verify the process without relying on a third party.

  • Based on the presence of smart contracts there are programmable and non-programmable blockchains .

    1. Programmable Blockchains
      • Contains smart contracts and an execution enviroment like Ethereum Virtual Machine (EVM) or WebAssembly (Wasm) to run the contracts.
      • Support development of Decentralized Applications (dApps) such as in DeFi apps.
      • Ethereum, Cardano, Tron, NEO
    2. Non-Programmable Blockchains
      • Focus on tasks such as securing transactions and data storage.
      • They use simpler logic and are not built for decentralized apps or automated agreements.
      • E.g) Bitcoin, Litecoin, Dogecoin, Monero

Blockchain Structure & Public Key Cryptography

  • Structure of a Blockchain
    • The data is stored in blocks
    • Each block contains:
      1. A List of Transactions
        • Ensures everyone agrees on who did what when.
      2. A Timestamp
        • The exact time and date when the block was created or added to the chain.
        • Helps maintain chronological order of the blocks.
      3. Hash Code
        • It represents the block by a unique string of letters and numbers.
        • It is generated by running the block data through a hash function.
        • Even a tiny change in the block's data will create a completely different hash.
        • Hence, it helps detect tampering, making the block content verifiable and secure.
      4. Hash Code of Previous Block
        • Links blocks together in order, forming a blockchain. This helps establish the exact order of transactions.
        • Ensures that the history of data is complete and unbroken If one block is altered, all subsequent blocks must also be modified to maintain consistent hash codes in the blockchain. — no one can remove or insert a fake block.
  • Public Key Cryptography
    • It is used to secure transactions and data on the blockchain using 2 keys:
      1. Public Key: Serves as an address for receiving cryptocurrency or data, and allows others to send funds or data to the correct address.
      2. Private Key: A confidential key that grants control over digital assets. The private key holder can authorize transactions, thereby providing security and verifying ownership.

Transactions & Transaction Models

  • Core Data in a Cryptocurrency Transaction
    1. Sender's Address: The wallet address of the person sending the crypto (derived from their public key).
    2. Recipient's Address: The wallet address of the person receiving the crypto ( derived from their public key).
    3. Amount: Amount of crypto being sent.
    4. Digital Signature: A cryptographic signature made The private key and transaction data is run through a digital signature algorithm (like Elliptic Curve Digital Signature Algorithm) to produce the signature. with the sender's private key, proving they authorized the transaction.
    5. Timestamp: The time the transaction was created.
    6. Transaction Fee: An optional (but often required) fee paid to miners or validators to process the transaction.
    7. Cryptographic Nonce (in some blockchains): A random or pseudo random number used in PoW blockchains (discussed later) to solve cryptographic puzzles. It is also used in some PoS networks (discussed later) to add randomness to the system and ensure it remains fair, unpredictable, and secure.
  • A transaction is formatted using a blockchain protocol A blockchain protocol refers to the rules and structure that define how a particular blockchain works — including how transactions are created, verified, stored, and added to blocks. .

  • Transaction Models : They define how a blockchains structure and processe transactions between users.
  • It determines:
    • How balances are tracked
    • How funds are transferred
    • How transaction history is recorded and verified
    1. Unspent Transaction Output (UTXO) Model
      • UTXOs It represent chunks of cryptocurrency that can be spent in future transactions.
      • Users utilizing this model will own a collection of UTXOs, which together determine their total balance.
      • A transaction in this system has:
        • One or more inputs (UTXOs being spent).
        • One or more outputs (new UTXOs created).
        • If the total value of input UTXOs is more than the required amount, change is sent back to the sender as a new UTXO.
        • For example: If you have $10 and $5 UTXOs and want to buy something worth $12, you must use both UTXOs (since they can't be split). You'll spend the $10 and $5, and receive a new $3 UTXO as change. After the transaction, you're left with only a $3 UTXO.
      • Once a UTXO is used as an input in a transaction, it is considered spent and cannot be reused.
      • Used by: Bitcoin, Litecoin, Dogecoin
    2. Account-Based Model
      • In this model, each user has an account that holds a balance, similar to a traditional bank account.
      • The account keeps track of: current balance, a nonce It represents the number of transactions an account has sent. The nonce helps prevent replay attacks—where someone tries to resend an old transaction—by ensuring each transaction has a unique, incrementing number. If a duplicated transaction uses an outdated nonce, the network rejects it. , other metadata such as a smart contract’s code (which defines its behavior) and the data it stores to keep track of information. .
      • A transaction in this model includes the sender's and receiver's addresses, the amount to be transferred, a signature proving the sender’s authority, and a nonce to ensure uniqueness.
      • When a transaction is executed:
        • The sender's balance is reduced by the amount (plus fees, if any).
        • The receiver’s balance is increased by the amount.
      • Unlike UTXOs, this model does not involve selecting or creating outputs. The state (account balances) is simply updated.
      • Used by: Ethereum, Solana, Binance Smart Chain

Adding a Transaction to a Blockchain (Part 1)

  • Transaction Creation: A transaction is initiated in a crypto wallet with the sender’s address, recipient’s address, and the amount to send.
  • Authorization: The sender signs the transaction with their private key, proving ownership and allowing the transfer to the recipient's public address.
  • Formatting: The transaction is structured according to the blockchain’s model — e.g., UTXO for Bitcoin, account-based for Ethereum.
  • Distribution to the Nodes Individual devices that participate in the blockchain network. :
    • The client software sends the signed transaction to a local node The nearest or directly connected node. . This node is like an entry point into the blockchain network.
    • This is followed by a peer-to-peer propagation of the transaction between connected nodes.
    • Nodes that receive the transaction place it into their memory pool (mempool) A waiting area for unconfirmed transactions after the they pass basic checks:
      1. Is the signature valid?
      2. Does the sender have enough balance?
      3. Does it follow all protocol rules (e.g., size limits in Bitcoin)?
    • Invalid or malformed transactions are discarded
    • The receiver’s wallet software is connected to its own set of blockchain nodes. When one of these nodes receives the transaction from the peer-to-peer network, it verifies the details and notifies the user of an incoming transaction with 0 confirmations (meaning it has not yet been included in a block — confirmations explained in block creation ).
  • Redundancy and Network Consistency
    • Because the network is decentralized, each node maintains its own mempool.
    • Over time, as the transaction spreads, most nodes will have a copy of it.
    • This redundancy ensures the transaction is not lost even if some nodes go offline.

Adding a Transaction to a Blockchain (Part 2)

  • Block Creators
    • Most blockchains have a special set of nodes to add new blocks to the chain.
    • Based on how blocks are created and validated, blockchain networks are broadly classified into two types: Proof of Work (PoW) and Proof of Stake (PoS) systems.
  1. Proof of Work: (Used by e.g., Bitcoin, Litecoin)
    • The special nodes are participants called miners who compete to create new blocks.
    • They must solve difficult cryptographic puzzles ( hash puzzles ) that require powerful hardware and significant electricity.
    • The first miner to solve the puzzle gets the right to add the new block to the blockchain.
    • When a miner successfully adds a block to the blockchain, he/she earns two types of rewards:
      1. Block reward: This is a new cryptocurrency that is created by the network and given to the miner. This is how new coins are introduced into circulation.
      2. Transaction Fees: Every transaction in the block includes a small fee, paid by the users who made those transactions. The miner collects all these fees as part of their reward.

  2. Proof-of-Stake (Used by e.g., Ethereum 2.0, Cardano):
    • A user who locks up (freezes) a certain amount of cryptocurrency as stake in the blockchain network to show commitment and qualify to participate in maintaining the network is a validator .
    • Energy consumption here is much lower than in PoW, since there's no need for solving energy-intensive puzzles.
    • These metworks are typically categorized into two types: those that assign validators two distinct roles, and those that assign only a single role.
    • Networks with Block Proposers and Attesters
      • These networks separate the role of proposing and attesting a block for validators. A proposer creates the block, while a committee of attesters verifies and finalizes it.
      • Selection Method: Stake-Based + Cryptographic Randomness
        • Validators stake a minimum amount of the native token.
        • At every time unit (slot) , one validator is randomly selected as the block proposer, and a committee of validators is randomly selected as attesters.
        • Selection is often done using cryptographic tools like Verifiable Random Function (VRF) to generate randomness.
        • In most stake-based systems (like Ethereum), the more you stake, the higher your probability of being selected as a proposer or attester.
      • Rewards:
        • Block Proposer: Receives newly minted coins + transaction tips/ entire transaction fee (depending on network). The proposer also gets a bonus in newly minted coins for including attestation votes from the committee (incentivizes inclusion of attestations in the block).
        • Attesters: Earn newly minted tokens for timely and accurate attestations (i.e., voting on block validity). They an be slashed Lose a portion of their staked tokens for dishonest behavior.
    • Networks with No Seperate Attesters
      • Validators take care of both proposing and validating blocks — no second group of attesters is used.
      • This network includes delegators: They are token holders who don’t run validator nodes themselves but stake their tokens by assigning (delegating) them to trusted validators, helping secure the network and earning a share of the rewards.
      • Importance of delegators: Since one’s own stake might not be enough to become an active validator, delegators are needed to vote/stake for a node.
      • Selection Methods:
        1. Rotation/ Round-Robin Selection
          • Token holders vote for top validators.
          • The top 'N' validators (e.g., 21) take turns proposing and signing blocks.
          • Every validator gets a fair share of turns over time.
        2. Delegated Voting
          • Token holders delegate votes to validator candidates.
          • Top-voted validators form the active set.
          • These validators both propose and validate blocks (no attesters)
      • Rewards:
        • Validators: Earn newly minted coins + transaction tips/ entire transaction fee (depending on network). They often share rewards with delegators.
        • Delegators: Get a portion of the validator's rewards (as per agreement or protocol rules).

Adding a Transaction to a Blockchain (Part 3)

  • Forking:
    • A fork is when the blockchain splits into two separate chains (two different versions of truth).
    • This can happen if two validators or miners create valid blocks at the same time, or due to delays in how blocks are propagated across the network.
    • Some nodes may receive one block first, while others receive a different one — leading to a temporary divergence.
    • The network continues building on both chains until one becomes longer. At that point, the longest chain is accepted, and the shorter one is discarded.
    • To ensure their transaction stays on the final chain, recipients wait for multiple confirmations (blocks built on top) before accepting it as final.
    • Transactions in the discarded chain are returned to the mempool.
    • These can be included in future blocks — unless the coins were already spent in a confirmed transaction on the main chain, in which case the original transaction becomes invalid and is dropped.
    • Creators (miners/ validators) of the discarded blocks do not recieve their rewards. This is because a reward is only considered final if a block becomes part of the accepted main chain.
  • Double Spending:
    • Double-spending occurs when someone attempts to use the same cryptocurrency in two different transactions.
    • This leads to a fork in the blockchain.
    • The blockchain only allows one of these transactions to be confirmed. The first one added to a block becomes valid, while the other is rejected.
    • To protect against this, recipients often wait for multiple confirmations (i.e., blocks added on top of the one containing their transaction). This makes it increasingly difficult to reverse or tamper with the confirmed transaction.
  • Consensus and Block Addition
    • Consensus is the process by which all nodes in a blockchain network agree on the same version of the blockchain — especially about which transactions and blocks are valid.
    • Consesus Algorithms
      1. PoW: Miners compete to solve a puzzle. The longest valid chain (with the most work) is accepted by the network.
      2. PoS: A validator is chosen to propose a block, and others vote (attest); if enough agree, the block is added.
      3. BFT- Style Protocols (Byzantine Fault Tolerant) (Used by: Tendermint (Cosmos), Diem (Meta)): Nodes communicate and must reach a 2/3 majority (typically) agreement before finalizing a block, ensuring fast and final consensus.
    • Once enough nodes agree, the block is added to the chain.
    • All nodes update their local copy of the blockchain.
    • Future blocks built on top of this one confirm its validity.
  • Transaction Settlement
    • After the block is added, the transaction becomes part of the permanent blockchain record. It cannot be changed or deleted.
    • The recipient officially receives the funds (or data is updated, in the case of non-currency uses).

Multiple Chains in a Blockchain

  • Apart from temporary forks, there are also intentionally designed multiple-chain blockchain systems.
  • Two common ones are:
  1. Multi-Chain Blockchains
    • It is a setup in which multiple blockchains operate within the same ecosystem, but each has its own ledger, rules, and purposes.
    • Each chain runs independently.
    • They may or may not talk to each other directly.
    • They are often used for scalability Splitting work across multiple chains so the system can handle more transactions at once without slowing down. or specialization Designing each chain for a specific job like payments & transfers, identity & authentication. .
  2. Cross-Chain Blockchains
    • The ability for two different blockchains to interact and exchange information or assets with each other.
    • They can communicate through built-in protocols Communication tools built directly into the blockchain system so chains can send data or tokens to each other safely, without needing extra services (like Cosmos’s IBC). , blockchain-bridges Apps that connect separate blockchains, often by locking assets on one chain and creating equivalent tokens on another. , or intermediary “hub” chains A central blockchain that works like a post office, receiving messages or transactions from one chain and delivering them to another (like Polkadot’s Relay Chain). to share data or transfer assets.
